## 1. 세션의 출발점 — 요청을 기획 언어로 해독하기

### 1.1 사용자 요청의 두 단계

| 단계 | 사용자 표현 (요지) | 기획적 해석 |
|------|-------------------|-------------|
| ① | “현대에서 최고 수준이 되게끔 **업그레이드 기획안**” | 단일 **마스터 플랜**: 비전·갭·로드맵·KPI·리스크·검증을 한 축으로 묶음 |
| ② | “할 것을 **차근차근 다** 진행” | 플랜을 **실행 가능한 스프린트 단위**로 쪼개 순차 검증 (본 보고서에서는 실행 세부 대신 **기획이 낳은 의무**만 기술) |

**왜 두 단계로 나누어 읽는가**  
①만 있으면 “문서 한 장”에 그칠 수 있고, ②만 있으면 **목표 함수 없이 작업 목록**이 된다. 세션 전체는 **①으로 방향을 고정하고 ②로 증거를 쌓는** 구조가 됐다.

### 1.2 이해관계자 렌즈 (기획 검토 시 누가 무엇을 묻는가)

기획안은 다음 **가상 질문**에 답할 수 있어야 한다. 이 질문들이 `14`번 문서의 목차와 표를 직접 설계했다.

| 페르소나 | 전형적 질문 |
|----------|-------------|
| **금융 도메인 담당** | 숫자·결론의 **출처**는 어디이고, 잘못됐을 때 **책임·재현**은? |
| **준법·감사** | 누가 무엇을 조회했는지, **로그**는 남는가, PII는? |
| **플랫폼·운영** | 장애 시 **복구**, 지연·비용은 **어디서 보는가**? |
| **제품·UX** | 사용자가 **언제 개입**하고, 진행 상황을 **어떻게 신뢰**하는가? |
| **개발 리드** | 스코프가 폭발하지 않게 **자르는 규칙**은? |

**어떻게 답하게 설계했는가**  
North Star 한 문장, NFR 수치표, citation·감사·SLO·HITL·스프린트 Exit, 리스크 레지스터를 **동시에** 둔다.

### 1.3 “최고 수준”을 조작적 정의로 번역

구어의 “최고”는 기획에서는 아래 **다섯 축**으로만 의미가 있다. 각 축은 `14`번 §2.1과 연결된다.

| 일상어 | 기획적 정의(조작적) | 마스터 플랜의 측정·근거 |
|--------|---------------------|------------------------|
| 정확하다 | 근거 있는 답, 환각 **입증 가능한 수준**으로 낮춤 | RAGAS faithfulness ≥ **0.85**, answer_correctness ≥ **0.80**, 골든셋 **30문항**/주 |
| 빠르게 느껴진다 | 첫 피드백·전체 시간에 **SLO** | TTFU p95 ≤ **1.5s**, E2E p95 ≤ **20s** |
| 추적 가능하다 | 결론마다 **인용 단위 ID** | 결론 **100%** chunk·node ID, citation 부착률 **100%** 목표 |
| 안 죽는다 | 단일 외부 의존 단절 시에도 **합리적 응답** | fallback 성공률 ≥ **99%**, chaos **6종** 통과(목표) |
| 회사에서 쓸 만하다 | **재현·배포·보안** | 단일 명령 dev/staging 재현, 인증·CORS·감사·시크릿 정책 |

**왜 번역이 필요한가**  
“SOTA”·“최고”는 회의실에서 합의가 안 된다. **숫자·통과 조건**으로 바꿔야 우선순위 싸움이 끝난다.

### 1.4 제품 한 문장 (기획의 압축 명제)

마스터 플랜 Executive Summary의 정의는, 이후 모든 기능에 대한 **헌법**이다.

> DART·공시 등 데이터를 **그래프와 벡터에 이중 인덱싱**하고, **근거(Evidence)에 묶인** 다중 에이전트가 **토론·검증·합의**하는 과정을 **추적·평가·재현** 가능한 형태로 **실시간 스트리밍**하는 **엔터프라이즈급 의사결정 엔진**.

**무엇을 강제하는가 (기획적으로)**  
- “인덱스가 두 종류” → 검색·추론의 **교차 검증** 전제.  
- “근거에 묶임” → 답만 내는 챗봇이 아닌 **DSS**.  
- “추적·평가·재현” → 감사·CI·릴리즈의 **증거 패키지**.  
- “실시간 스트리밍” → UX·운영이 **시간축 데이터**를 소비.

**왜 한 문장인가**  
스프린트 중 “이 기능 넣자”가 나오면 **이 문장을 약화시키는지**만 보면 된다.

### 2.2 API·스트리밍 계약의 진화 (`4_Backend_API_and_SSE.md` RFC)

**무엇을**  
SSE **v1 호환 유지** 전제 하에, **v2 = `type` 기반 이벤트** (node_start, token, evidence_added, node_end, interrupt_requested, done, error 등).

**왜**  
- 프론트·백·외부 연동 주체가 **같은 스키마表**를 봐야 분쟁이 줄어든다.  
- “노드 JSON 한 덩어리”만으로는 **토큰 체감·근거 도착·개입** 스토리를 표현하기 어렵다.

**어떻게 (기획 프로세스)**  
**확장(Extension)** 이라는 정책 — v1 깨면 기존 데모·문서 전부가 부정당하므로, **병행 송출**을 전제로 RFC를 올렸다.

**왜 병행인가**  
하위 호환은 **기술 부채 방지**가 아니라 **조직 부채 방지**다 (갑자기 클라이언트 전부 수정 불가).

## 3. North Star · NFR · 정성 비전 (구체 수치)

### 3.1 5대 NFR (마스터 플랜 §2.1 그대로의 기획 의미)

| NFR | 임계값 | 측정 방법 | 왜 이 숫자를 말하는가 |
|-----|--------|-----------|------------------------|
| **정확성** | Faithfulness ≥ **0.85** | 골든셋 30문항/주 | 환각 **입증** 없이 금융 의사결정 엔진 불가 |
| **응답성** | TTFU p95 ≤ **1.5s**, E2E p95 ≤ **20s** | OTel histogram | “생각하는 중” 피드백·전체 대기 **체감 계약** |
| **추적성** | 결론 **100%** 인용 가능 ID | citation 검증 lint | **규제·내부감사** 대응 |
| **회복력** | 단일 의존 단절 시 fallback ≥ **99%** | chaos test | 외부 API는 **항상 깨질 수 있음**을 전제 |
| **운영성** | 1명·1명령 재현 | compose / make | 인력 교체·장애 재현 **비용 절감** |

**재차: 왜 faithfulness 0.85인가**  
0.9는 이상적이지만 PoC 단계에서 달성 불가 시 **팀 사기·허위 리포트** 위험. 0.85는 **도전적이되 합의 가능한 1차 계약선**으로 기획했다.

### 3.2 정성 비전 (마스터 플랜 §2.2 — UX·운영 스토리)

> 사용자는 **한 줄 질문** → **1.5초 내** 첫 사고 흐름 → 에이전트 **다양한 의견** → 비판자가 **근거 요구** → **추가 검색** → **합의** → **클릭 가능한 출처** → 사용자는 **중단·변수 변경·재개** → 운영자는 **토큰·비용·지연·실패**를 시각화.

**이 비전이 기획에 미치는 강제력**  
- SSE에 **token·evidence_added·interrupt**가 필요한 이유.  
- Memory에 **thread resume**이 필요한 이유.  
- Ops에 **비용·지연 메트릭**이 필요한 이유.

## 4. As-Is → To-Be 와 갭 분석 (7 × 다수 항목의 기획 논리)

### 4.1 9축 요약 표 (기획 토의용)

마스터 플랜 §0.2의 정신을 유지해, **의사결정 단위**로 압축한다.

| 축 | As-Is (기획적 한계) | To-Be (성숙 목표) |
|----|---------------------|-------------------|
| 검색 | dense 중심, **분해·rerank·커뮤니티** 약 | Hybrid + 분해 + (선택) rerank + 안전 Cypher + 요약 |
| 에이전트 | 직선형, **비판·반성** 약 | Intent → 검증 가능한 **토론·재검색 루프** |
| 토론 | 합의가 형식적일 수 있음 | **disagreement**·**최대 반복**·명시적 합의 실패 |
| 근거 | 서술 위주 | **evidence_id 의무**, citation 그래프 방향 |
| 메모리 | 프로세스 수명 | **체크포인트·에피소드·(장기) 메모리** |
| LLM | 단일 호출 패턴 | Router·캐시·예산·가드·통일 재시도 |
| 평가 | 소수 테스트 | **골든셋·RAGAS·적대적·CI 게이팅** |
| 운영 | 수동 | 관측·컨테이너·파이프라인 |
| 보안·UX | 데모 설정 | 인증·감사·**HITL·가시화** |

### 4.2 갭 카테고리 A~G (왜 이렇게 쪼갰는가)

마스터 플랜 §1.2는 갭을 **A Retrieval ~ G UX**로 나눈다. 이는 “기능 목록”이 아니라 **실패 모드 목록**이다.

| 카테고리 | 대표 실패 모드 (기획적) | 기획적 대응 축 |
|----------|-------------------------|----------------|
| **A. 검색** | 회수율 낮음, multi-hop 단일 쿼리 처리 | Planner, Hybrid, Rerank, Community |
| **B. 에이전트** | 비판 없음, 도구 일반화 없음 | Critic, Reflector, Tool 표준 |
| **C. 메모리** | 재시작 시 상태 소실 | PostgresSaver, Resume API |
| **D. 평가·관측** | 환각·지연 불가시 | RAGAS, OTel, SLO |
| **E. 신뢰성** | 외부 장애가 UX를 장애로 | Circuit breaker, retry 단일화, chaos |
| **F. 보안** | 무인증, 로그 없음 | JWT/OAuth, audit, PII mask |
| **G. UX** | 근거·개입 불가 | Drawer, graph, HITL, a11y |

**왜 38개 세부 항까지 내려가는가**  
스프린트 계획 회의에서 “이번에 뭐 하지?”를 **주관**으로 하지 않기 위해. 리스트는 **이미 합의된 backlog** 역할.

### 4.3 갭 분석 방법론 (프로세스)

1. **자산 먼저** (§1.1 강점): “전부 갈아엎기” 방지, **이식 비용** 절감.  
2. **실패 모드 단위로 쪼개기**: 기능이 아니라 “**무엇이 깨지면 누가 화내는가**”.  
3. **Pillar에 매핑**: 여러 갭이 한 Pillar에 모이면 **시너지 스프린트** 가능.  
4. **Exit 기준에 연결**: 갭 닫음 = 스프린트 종료 조건.

## 5. 7대 Pillar — 기획 수준 상세 (목적·방법·검증·의존)

각 Pillar는 **왜 단독으로 존재하는가**와 **다음 Pillar에 무엇을 넘기는가**를 명시한다.

### Pillar 1. Retrieval 2.0

**무엇을**  
질의 **이해·분해**, 하이브리드 검색, (선택) rerank, **안전한** 그래프 질의, 커뮤니티 요약, 캐시, **citation 단위 evidence**.

**왜**  
금융 DSS에서 **검색이 틀리면** 이후 에이전트는 아무리 똑똑해도 **정교한 거짓**을 만든다. 에이전트 투자 전에 검색 성숙도를 올리는 것이 **ROI**가 크다.

**어떻게 (기획 메커니즘)**  
1. **Query Planner**: sub-query 1~5개, 엔티티·연도·의도 태깅.  
2. **Hybrid**: dense + sparse 가중, α를 의도별 동적 조정(기획안에 명시).  
3. **Rerank**: K=20 → 5 압축.  
4. **Cypher**: LLM 합성 + **read-only 화이트리스트** (보안 사고 방지).  
5. **Community summary**: multi-hop 시 상위 요약 → drill-down.  
6. **Caching**: 임베딩·(향후) 시맨틱 답변 캐시.

**검증(Exit 예시)**  
- multi-hop 골든 **5건** recall@5 ≥ **0.9**  
- 캐시 적중 ≥ **30%** (반복 워크로드)  
- p95 retrieval ≤ **1.2s**

**의존**  
Pillar 2의 “인용 강제”는 Pillar 1의 **evidence_id**가 있어야 성립.

### Pillar 2. Agent Mesh 2.0

**무엇을**  
Intent 분류, (개념상) Planner·Retriever, 분석가, **Critic**, **Reflector**, Orchestrator, **HITL** 분기.

**왜**  
직선 파이프라인은 “**연극 같은 합의**”에 그칠 수 있다. 금융 의사결정에는 **이견·검증·재검색**이 기본값이어야 한다.

**어떻게 (기획 규칙)**  
- `MAX_TURNS`, `MAX_REFLEXIONS`, `disagreement_score ≤ 0.2` 시 합의 같은 **원자적 라우팅 규칙**.  
- **예산·시간 회로 차단**으로 비용 폭주 방지.  
- 모든 구조화 출력에 **evidence_ids 의무**.

**검증(Exit 예시)**  
- 시나리오 **A**(자회사 리스크 전이): critic이 **최소 1회** 재검색 트리거.  
- 시나리오 **B**(상충 데이터): `MAX_TURNS` 도달 시 **명시적 합의 실패**.  
- evidence 미부착 응답 **0건**.

**의존**  
Pillar 1의 검색·인용 없이는 Critic이 할 일이 없다.

### Pillar 3. Memory & State

**무엇을**  
Short-term **checkpointer**(Postgres 권장), **episode** 아카이브, (선택) long-term vector memory, **스키마 버전**, Resume/Interrupt **제품 스토리**.

**왜**  
의사결정은 한 세션에 끝나지 않는다. **재현·감사·사용자 개입** 모두 “상태가 남는다”는 전제가 필요하다.

**검증(Exit 예시)**  
- 서버 재기동 후 동일 `thread_id`로 **동일 결과 재현**.  
- episode **N일** 적재 후 “이전 분석” 신호 활용(기획 방향).

### Pillar 4. LLM Layer

**무엇을**  
Model Router(추론 vs 추출), Provider 추가, 시맨틱 캐시, **예산**, 입·출력 **가드**, 통일 **재시도·타임아웃**.

**왜**  
모델·벤더는 **가격·정책·장애**가 변한다. 단일 `httpx` 호출은 **운영 사각지대**를 만든다.

**검증(Exit 예시)**  
- 동일 질의 반복 시 캐시 적중 **≥ 95%** (마스터 플랜 제시).  
- p95 reasoning ≤ **8s**.  
- prompt-injection **30종** 차단 ≥ **90%**.

### Pillar 5. Data & Ingestion

**무엇을**  
DAG·증분 적재·병렬 임베딩·스키마 마이그레이션·**품질 게이트**·**lineage** 메타.

**왜**  
RAG 품질 상한은 **데이터 품질 상한**이다. 엔진만 올리지 않고 적재를 올리면 **환각이 아니라 데이터 버그**가 된다.

**검증(Exit 예시)**  
- 5사×4년 등 **시간·병렬화** 목표(예: ≤ **5분**).  
- quality gate 위반 **0건**일 때만 alias 스왑.

### Pillar 6. UX / Frontend 2.0

**무엇을**  
3-pane, **토큰** 스트림, Evidence Drawer, 라이브 KG(Cytoscape/React Flow **1종**으로 한정), Debate timeline, HITL panel, **a11y·i18n**.

**왜**  
내부 애널리스트는 “답”이 아니라 **근거·반론·개입**을 동시에 봐야 신뢰한다.

**검증(Exit 예시)**  
- TTFU p95 ≤ **1.5s**.  
- Lighthouse a11y ≥ **95**, 모바일 perf ≥ **80**.

### Pillar 7. Ops · Eval · Security

**무엇을**  
OTel, Prometheus, Grafana, 구조화 로그, **골든셋+RAGAS+적대적**, CI/CD, **Auth**, Vault류 시크릿, CORS 화이트리스트, rate limit, **audit**, Trivy/Snyk 등.

**왜**  
엔터프라이즈에서 “작동 데모”와 “**도입 승인**”의 차이는 **관측·보안·릴리즈 게이트**다.

**검증(Exit 예시)**  
- Trace에서 단일 요청 **5단계 이상** drill-down.  
- chaos **6종** sane fallback.  
- 이미지 **high CVE 0건** 목표.

### Pillar 간 의존 그래프 (기획 요약)

```
데이터(P5) ──► 검색(P1) ──► 에이전트(P2) ──► 메모리(P3)
                     ▲              │
                     │              ▼
                 LLM(P4) ◄──── UX(P6) ◄── Ops/Sec(P7) 관측·게이트
```

**왜 이 순서로 읽는가**  
데이터·검색이 어긋나면 나머지는 **성능 튜닝이 아니라 연극**이 된다.

|------|------------|
| **North Star** | 우선순위 필터가 되는 한 문장 목표 |
| **NFR** | 기능이 아닌 품질·비용·신뢰 요구 |
| **Pillar** | 능력 레이어(검색·에이전트·…) |
| **Exit** | 스프린트 종료 시 **합의된 증거** |
| **게이팅** | 머지/배포를 막는 **자동 조건** |
| **HITL** | Human-in-the-loop, **개입·재개** 제품 스토리 |

---

*끝. 본 확장판은 기획 내용만을 극대화했으며, 구현 세부·커밋 해시는 포함하지 않는다.*
