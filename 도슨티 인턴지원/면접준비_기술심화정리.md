```
사용자 질의
    ↓
[Guardrails] 프롬프트 인젝션·범위 외 질의 차단
    ↓
```

## 1. LangGraph — 왜 단순 체인 대신?

**선택 이유 3가지:**

1. **State Machine 방식**: 각 노드가 `GraphState`라는 공유 상태를 읽고 쓰는 구조. 어느 단계에서든 상태를 저장·복원 가능 → 중단된 분석 재개 가능 (Checkpointing).

2. **Conditional Routing**: `critic_node`가 품질 점수를 계산하면 `router_logic`이 다음 노드를 결정. 단순 LLM 체인은 이 분기 로직을 외부에서 구현해야 하지만, LangGraph는 `add_conditional_edges()`로 선언적 처리.

3. **자가 교정 루프**: critic → reflector → retrieve_context로 되돌아가는 사이클이 LangGraph의 핵심 강점. 단순 체인은 역방향 순환 불가.

**LocalFallbackGraph 설계 이유:**
```python
class LocalFallbackGraph:
    # LangGraph 미설치 환경(면접 노트북 등)에서도 동일 동작 보장
    # 안전가드: max_steps = (MAX_TURNS + MAX_REFLEXIONS) * 4 → 무한루프 방지
```

## 2. 하이브리드 RAG = Dense + Sparse + Graph

단순 벡터 검색(dense only)의 한계를 세 가지 전략으로 보완:

### ① Dense (Qdrant + Gemini 임베딩)
```
텍스트 → Gemini embedding-001 → 768차원 벡터 → Qdrant 코사인 유사도 검색
```
- **강점**: 의미 유사성 (동의어, 문맥 이해)
- **약점**: 정확한 수치나 고유명사 매칭에 취약

### ② Sparse (BM25-Lite, `sparse.py`)
```
한국어 토크나이저: 공백 어절 + 한글 음절 bigram → TF-IDF 스타일 점수
```
- **강점**: "부채비율", "HBM3E" 같은 정확한 키워드 매칭
- **의존성 0**: 외부 BM25 라이브러리 없이 자체 구현 (rank_bm25 있으면 자동 사용)
- **왜 bigram?**: 한국어는 "삼성전자"가 "삼성"+"전자"로 분리돼야 검색되므로 음절 bigram으로 회수율 보강

### ③ Graph (Neo4j 2-hop 탐색)
```
카카오 → [HAS_REPORT] → FinancialReport
카카오 → [INVOLVED_IN] → Disclosure(카카오게임즈 규제)
      2-hop: 카카오게임즈 → [INVOLVED_IN] → Disclosure
```
- **강점**: "카카오게임즈 리스크가 카카오 본사에 미치는 영향" 같은 관계형 질의
- 벡터 검색만으로는 절대 못 찾는 엔티티 간 연결 관계를 커버

### 융합 전략 (reranker.py)
```python
final_score = alpha × dense_score + (1-alpha) × sparse_score
# + payload_richness 보정 (revenue, debt_ratio 등 값이 있는 필드 수로 가중)
# + lexical_overlap 보정 (쿼리 토큰 ∩ 문서 토큰)
# alpha는 query intent에 따라 동적 조정 가능 (planner가 힌트 제공)
```

## 3. QueryPlanner — 질의 분해 전략

**LLM이 있을 때**:
- Gemini에게 JSON 출력 요청 → `SubQuery` 1~5개 분해
- 각 SubQuery에 `intent`, `target_company`, `target_year`, `weight` 부여

**LLM이 없을 때 (휴리스틱 fallback)**:
```python
# 1) 다중 기업 쿼리: 회사별 SubQuery 분리
#    "삼성전자와 SK하이닉스 비교" → SubQuery("삼성전자: ..."), SubQuery("SK하이닉스: ...")
# 2) 단일 기업: 문장부호·접속사 기준 절 단위 분해
# 3) intent 분류: 키워드 매칭 (trend/relation/risk/facts)
```

**캐시**: 동일 질의는 10분 TTL LRU 캐시에서 재사용 → Gemini API 비용 절감

## 4. Critic 노드 — 품질 게이트 상세

**환각 탐지 3가지 기준:**

| 기준 | 감점 | 탐지 방법 |
|------|------|----------|
| 인용 누락 | +0.4 | `finance_metrics.evidence_ids`가 비어 있음 |
| 증거 없음 | +0.2 | `state["evidence"]` 리스트가 비어 있음 |
| 모순 탐지 | +0.4 | debt_ratio < 150인데 insight에 "상승" 포함 등 |
| 데이터 품질 경고 | +0.1 | `financial_parse_failed` 플래그 |

**재검색 트리거 조건 (AND):**
```python
request_re_retrieval = (
    score >= 0.5                           # 품질이 나쁘고
    and reflexion_count < MAX_REFLEXIONS   # 아직 재시도 횟수 남았고
    and not has_evidence                   # 증거가 아예 없을 때
)
```

## 5. Circuit Breaker — 외부 서비스 장애 처리

Qdrant·Neo4j 클라우드가 다운됐을 때 전체 파이프라인이 멈추는 것을 방지:

```
closed(정상) → [연속 3회 실패] → open(차단, 즉시 CircuitOpenError)
open → [15초 대기] → half_open(1회 테스트 허용)
half_open → [성공] → closed
half_open → [실패] → open (다시 차단)
```

**코드:**
```python
QDRANT_BREAKER = CircuitBreaker(name="qdrant", failure_threshold=3, recovery_time_s=15.0)
NEO4J_BREAKER  = CircuitBreaker(name="neo4j",  failure_threshold=3, recovery_time_s=15.0)
```

외부 서비스 실패 시 자동으로 seed 데이터 fallback 경로로 전환.

## 6. SSE 실시간 스트리밍 — 왜 WebSocket이 아닌가?

**SSE 선택 이유:**
- 단방향(서버→클라이언트) 스트리밍으로 충분
- HTTP/1.1 기반 → 방화벽 친화적, 별도 업그레이드 핸드셰이크 불필요
- FastAPI 기본 `StreamingResponse`로 구현 가능 (추가 라이브러리 없음)

**토큰 스트리밍 구조:**
```python
# orchestrator_node 내부:
for token in decision.split(" "):
    put_token(thread_id, "orchestrator", token + " ")
    # → asyncio.Queue에 적재

# endpoints.py 에서 Queue를 polling → SSE 이벤트로 전송
# data: {"event": "token", "node": "orchestrator", "delta": "재무"}
```

## 7. LLMRouter — 의도별 Provider 선택

```python
KNOWN_INTENTS = {
    "finance_metrics",  # 재무 분석
    "risk_points",      # 리스크 분석
    "planner",          # 쿼리 분해
    "critic",           # 품질 평가
    "intent_classifier",
    "reflexion",
```

- API 키 있음 → `OpenAICompatProvider` (Gemini를 OpenAI 호환 엔드포인트로 호출)
- API 키 없음 → `MockProvider` 자동 fallback (rule-based 결과 반환)
- **왜 OpenAI compat?**: Gemini가 `/v1/chat/completions` 호환 엔드포인트를 제공 → 동일 코드로 OpenAI/Gemini 교체 가능

## 9. 데이터 품질 플래그 시스템

```
DART 수집 시 생성:
  revenue_missing, operating_profit_missing
  debt_ratio_unavailable (자본 ≤ 0)
  financial_parse_failed (예외 발생)
  disclosure_not_found
  current_year_partial_data

→ Qdrant payload에 함께 저장
→ retrieve_context가 analysis_context.data_quality.flags에 포함
→ critic이 이 플래그를 읽어 disagreement_score 산정
→ LLM 프롬프트에 주입 ("flags에 missing/failed 있으면 단정 금지")
```

## 10. 기술 선택 근거 한눈에

| 기술 | 대안 | 선택 이유 |
|------|------|----------|
| **LangGraph** | LangChain Chain | 순환 그래프(critic→reflector→retrieve 루프)가 필요 |
| **Qdrant** | Pinecone, Weaviate | 자체 호스팅 가능, Rust 기반 고성능, payload 필터 |
| **Neo4j** | Amazon Neptune | Cypher 쿼리 직관적, 2-hop 관계 탐색 native 지원 |
| **Gemini embedding-001** | OpenAI text-embedding | 한국어 768차원, DART API와 동일 구글 에코시스템 |
| **FastAPI** | Flask, Django | async native, Pydantic 통합, SSE 구현 용이 |
| **SSE** | WebSocket | 단방향 스트리밍으로 충분, HTTP 기반 방화벽 통과 |
| **Circuit Breaker** | 단순 try/except | 연속 실패 시 빠른 차단 + 자동 복구 (timeout 누적 방지) |
| **BM25-Lite (자체 구현)** | rank_bm25 라이브러리 | 의존성 0 원칙, 한국어 음절 bigram으로 회수율 보강 |
| **OpenAI compat API** | 직접 Gemini SDK | Provider 교체 시 코드 변경 최소화 |

## 면접 핵심 3줄 요약

1. **단순 RAG와의 차별점**: 벡터(의미) + BM25(정확도) + 그래프(관계) 세 가지를 reranker로 융합한 하이브리드 구조. 단독 벡터 검색 대비 관계형 질의에서 월등한 recall.

2. **환각 제어 레이어**: Guardrails(입력 검증) → Critic(출력 품질 측정) → Reflector(자가 교정) → MAX_REFLEXIONS(무한루프 방지) 4중 방어.

3. **운영 견고성**: Circuit Breaker로 외부 DB 장애 격리 + Fallback-first 설계로 키 없는 환경에서도 파이프라인 전체 동작.
