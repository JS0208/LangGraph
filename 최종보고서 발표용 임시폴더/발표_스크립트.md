## 슬라이드 3 — 문제 인식

기존 AI 금융 분석 도구에는 세 가지 핵심 한계가 있습니다.

첫째, **관계 추론 실패**입니다. 기존 Vector RAG는 단순 유사도 검색만 합니다.
"카카오게임즈 리스크가 카카오 본사에 미치는 영향"처럼 기업 간 연결 관계를 추적하지 못합니다.

둘째, **교차 검증 부재**입니다. 단일 LLM은 자신의 환각을 스스로 감지하지 못해 금융 의사결정에 치명적입니다.

셋째, **근거 불투명**입니다. "어디서 나온 답인가"를 추적할 수 없어 투자자나 애널리스트가 결론을 신뢰하기 어렵습니다.

이 세 문제를 GraphRAG 관계 추론, 멀티에이전트 교차 검증, Evidence ID 의무 부착으로 해결했습니다.

## 슬라이드 4 — NFR (5대 비기능 요구사항)

프로젝트 목표는 DART 공시 기반 자연어 질의를 GraphRAG와 멀티에이전트로 처리하는 것입니다.

품질 목표를 수치로 정의했습니다.

- **정확성(Faithfulness)**: 목표 0.85 이상 → 실측 1.0으로 초과 달성했습니다.
- **응답성(TTFU p95)**: 목표 1.5초 이하 → SSE 스트리밍을 구현해 실시간 전송은 되나, 정확한 수치 측정은 진행 중입니다.
- **추적성(Citation 부착률)**: 목표 100% → 100% 달성했습니다.
- **회복력(Fallback 성공률)**: 목표 99% → Fallback-First 아키텍처로 DB 없이도 시연 가능하게 구현했습니다.
- **운영성**: docker compose up 단일 명령 재현 → GitHub Actions CI와 함께 달성했습니다.

## 슬라이드 6 — 전체 아키텍처

시스템은 5개 계층으로 구성됩니다.

- **클라이언트**: Next.js 16, 3개 패널 대시보드, SSE 실시간 스트리밍
- **FastAPI 레이어**: API 엔드포인트, JWT 인증, 감사 로그, 비동기 큐
- **LangGraph 오케스트레이터**: 8개 에이전트 파이프라인
- **데이터 레이어**: Neo4j 그래프 DB와 Qdrant 벡터 DB 하이브리드
- **관측성**: Prometheus 메트릭, OTel 추적, 구조화 로그, GitHub Actions CI

핵심은 **모든 외부 의존성에 Fallback 경로가 존재**해, DB나 LLM 없이도 완전 시연이 가능하다는 점입니다.

## 슬라이드 7 — GraphRAG 파이프라인

데이터 적재는 DART API에서 공시를 가져와 LLM으로 파싱, 기업 엔티티를 추출해 Neo4j와 Qdrant에 저장합니다.

검색 시에는 사용자 질의에서 기업명과 연도를 추출한 뒤, Neo4j 2-hop 그래프 검색과 Qdrant 하이브리드 벡터 검색을 병렬로 실행하고 결과를 합쳐 8개 에이전트에 넘깁니다.

핵심 가치는 **2-hop 관계 추론**입니다.
"카카오게임즈 → 지분관계 → 카카오 본사 → 재무 영향"처럼 단순 벡터 검색으로는 불가능한 연결 추적이 가능합니다.

---

## 슬라이드 8 — 8-Agent 오케스트레이션

8개 에이전트가 Intent → Retrieve → Finance → Risk → Critic → Reflector → Orchestrator → Report 순서로 처리합니다.

Critic과 Reflector가 결과를 재검토하는 **Reflexion Loop**를 최대 2회 실행합니다.
테스트 결과, 카카오게임즈 리스크 시나리오에서 Critic이 재검색을 1회 트리거했고, 합의 실패 시 명시적 합의실패 결론을 반환하는 것도 검증했습니다.

---

## 슬라이드 9 — Backend (FastAPI · SSE)

분석 요청 흐름입니다.

1. 사용자가 질의를 보내면 POST /start로 세션이 시작되고 thread_id가 발급됩니다.
2. GET /stream으로 에이전트 진행 상황이 실시간 SSE 스트림으로 전송됩니다.
3. 사용자가 중단 버튼을 누르면 POST /interrupt로 안전하게 태스크가 정리됩니다.

안정성 측면에서, 에이전트 하나가 30초를 초과하면 Fallback 이벤트를 내보내고 graceful하게 종료됩니다.

## 슬라이드 10 — 보안 4중 구조

보안은 4겹으로 구성됩니다.

- **JWT 인증**: 모든 API에 HS256 Bearer 토큰 인증을 적용했습니다.
- **Guardrail**: 프롬프트 인젝션 패턴을 감지해 플래너 단계에서 차단합니다.
- **입력 Sanitize + PII 마스킹**: 주민번호 같은 개인정보는 out_of_scope로 처리합니다.
- **감사 로그**: 모든 요청을 삭제 불가능한 append-only 방식으로 기록합니다.

현재는 JWT를 자체 발급하며, 향후 OAuth2 PKCE와 외부 IdP 연결을 예정하고 있습니다.

**Heuristic Gate**는 외부 LLM 없이 항상 실행 가능한 규칙 기반 평가로, GitHub Actions CI에 통합되어 pass_rate 75% 이상을 기준으로 합니다.

## 예상 질문 대비

**Q. Fallback이 구체적으로 어떻게 동작하나요?**
A. DB나 LLM 연결이 실패하면 사전 정의된 응답을 반환합니다. 모든 외부 의존성마다 Fallback 경로를 별도로 구현해 의존성 없이도 전체 플로우가 동작합니다.

**Q. 골든셋 33케이스는 어떻게 만들었나요?**
A. 직접 설계했습니다. 정상 케이스뿐 아니라 적대적 입력(jailbreak 시도), PII 포함 입력, 범위 외 질문까지 7개 클래스로 구성해 실제 서비스 환경을 가정했습니다.

**Q. TTFU p95 측정을 못 한 이유는?**
A. SSE 스트리밍 구현 자체는 완료됐지만, p95 분위수 측정을 위한 부하 테스트 자동화가 미완 상태입니다. 단기 로드맵에 포함되어 있습니다.

**Q. 5번과 14번 슬라이드가 비슷해 보이는데요?**
A. 의도된 구성입니다. 5번은 발표 초반 전체 개발 여정을 개관하고, 14번은 평가 결과 뒤에 위치해 "이 수치가 얼마나 체계적인 과정에서 나왔는지"를 뒷받침하는 역할입니다.
