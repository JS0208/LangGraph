## 0. 서치로 확인한 핵심 사실

**[현대차 제품 방향]**
- 현대차그룹 차세대 인포테인먼트 시스템 = **Pleos Connect** (2026년 Q2 출시, 2030년까지 2000만 대)
- AAOS(Android Automotive OS) 기반, 클라우드 인프라·Fleet Management 통합 E2E 플랫폼
- 내장 AI 어시스턴트 = **Gleo AI** — LLM 기반, 맥락 인식·다중 명령 동시 처리
- → **지성 씨 LangGraph 프로젝트(LLM 오케스트레이션)와 직접 연결 가능한 유일한 지원자**

**[자소서 형식]**
- 1페이지 자유형식 자소서는 **올해 처음 도입** — 합격 레퍼런스 없음 = 형식 자유도 높음
- 채용 담당자 공통 평가 기준: "직접 진행한 프로젝트 + JD 경험 매핑" > 정량 스펙

**[레이아웃]**
- 채용 담당자 모바일 검토 비중 높음 → 가로 2단 레이아웃 위험 → **세로 단단 구조 확정**
- GitHub/포트폴리오 링크 첨부 = 2025~2026 개발자 채용 필수 요소

## 1. 포지션 분석 — 이 공고가 원하는 것

**서버 백엔드 개발** 핵심 요구:
- RESTful/gRPC API 기반 커넥티드 카 서비스 백엔드
- 대규모 트래픽 처리, 글로벌 게이트웨이 서버 아키텍처
- 마이크로서비스 기반 실시간 알림 시스템 고도화
- 지능형 모니터링 서비스

**우대사항 (서버 백엔드):**
- 컴퓨터공학 전공 ✅
- Java/Kotlin/Go 기반 백엔드 경험 → **나는 Python. 언급하지 않는다.**
  동일한 아키텍처 패턴을 Python으로 구현한 것이 증거.

## 6. 문제 해결 사례 후보

**후보 A — SSE 비동기 스트리밍 설계** ← 1순위 추천
- 상황: 멀티에이전트 처리 중 노드별 중간 결과를 클라이언트에 실시간 전달 필요
- 선택: SSE + asyncio 비동기 큐, ping 유지·30초 타임아웃·취소 인터럽트 직접 구현
- JD 직결: "마이크로서비스 기반 실시간 알림 시스템 고도화"
- Pleos 직결: 차량-클라우드 실시간 통신 패턴

**후보 B — Prometheus + OTel 모니터링 설계**
- 상황: 노드별 지연·오류·트래픽 추적 없이는 시스템 운영 불가
- 선택: Prometheus 메트릭 + OTel 분산 추적 직접 구현, 히스토그램 버킷 설계
- JD 직결: "지능형 모니터링 서비스 개발"
- Pleos 직결: Fleet Management 관제 시스템

> A 또는 B 하나만 선택. 둘 다 쓰면 분량 초과.
